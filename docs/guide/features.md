## ReAct エージェントループ

LunaCode のコアとなるエージェントは **ReAct (Reasoning + Acting)** パターンで動作します。

### 動作の流れ

```
ユーザーのクエリ
   ↓
[THINKING] LLM がタスクを分析し、次のアクションを決定
   ↓
[ACTING]  選択したツールを実行（bash, read_file, edit_file 等）
   ↓
[OBSERVING] ツールの出力を LLM にフィードバック
   ↓
（ツールコールがなくなるまで繰り返し）
   ↓
最終回答をユーザーに返却
```

### 設定パラメータ

| パラメータ | デフォルト | 説明 |
|---|---|---|
| `maxIterations` | 50 | 1クエリあたりの最大ループ回数 |
| `timeout` | 15000ms | 各ツール実行のタイムアウト |

ループ終了後、メモリシステムの MicroCompact が自動実行され、コンテキストを圧縮します。

### コンテキスト管理

各ループ開始時に以下のコンテキストが LLM に渡されます:

- メインメモリ (`MEMORY.md`) の内容
- クエリに関連するメモリ検索結果（関連度スコアリング）
- 直近7日間のログ
- 利用可能なツールの定義

これにより、LLM は過去の作業履歴を参照しながらタスクを遂行できます。

## 3層セルフヒーリングメモリ

LunaCode のメモリシステムは3つのレイヤーで構成されています。

```
┌─────────────────────────────┐
│  Layer 1: MEMORY.md          │  ← 常にロード（ポインタ + 要約）
├─────────────────────────────┤
│  Layer 2: topics/*.md        │  ← オンデマンド読み込み
├─────────────────────────────┤
│  Layer 3: logs/*.log         │  ← grep 検索のみ
└─────────────────────────────┘
```

**Layer 1 (MEMORY.md)** — エージェントが常に参照するメインメモリ。プロジェクトの概要、重要な決定事項、トピックへのポインタを保持します。

**Layer 2 (topics/)** — トピック別に分類された詳細情報。architecture, features, bugs, tasks 等のカテゴリに自動分類されます。

**Layer 3 (logs/)** — 日付別の生ログ。ツール実行の記録が時系列で保存されます。検索時のみアクセスされます。

### メモリ検索

メモリ検索は3つのレイヤーすべてを横断して行われます。各結果は関連度スコアでランキングされ、上位の結果がエージェントのコンテキストに注入されます。

```bash
lunacode memory search "認証 API"
```

### MicroCompact

小規模なメモリ圧縮処理です。エージェントループの終了時に自動実行されます。

処理内容:
1. 重複行の削除
2. 空行の除去
3. ヘッダー行の保持
4. `maxContextLines` を超える行の切り詰め

### AutoCompact

大規模なメモリ再構成処理です。メモリの行数が `autoCompactThreshold`（デフォルト: 500行）を超えると自動でトリガーされます。

処理内容:
1. メモリ内容をトピック別に自動分類（キーワードベース）
2. 重要なセクション（ヘッダー行とその直後の内容）を抽出
3. 分類結果をトピックファイルとして保存
4. メインメモリをトピックへのポインタ + 重要情報のみに圧縮

### サーキットブレーカー

メモリシステムには障害耐性のためのサーキットブレーカーが実装されています。

- 連続5回のファイルアクセス失敗でサーキットが **OPEN** に遷移
- OPEN 状態では30秒間、メモリアクセスをスキップ（エラーメッセージを返却）
- 30秒後に **HALF-OPEN** → 成功すれば **CLOSED** に復帰
- 成功するたびに失敗カウンタがデクリメント

## KAIROS デーモン

KAIROS は LunaCode のバックグラウンド常駐プロセスです。

### Tick システム

60秒間隔で以下の TickTask が優先度順に実行されます:

| タスク | 優先度 | 説明 |
|---|---|---|
| ヘルスチェック | HIGH | システム全体の稼働状態を確認 |
| プロアクティブチェック | HIGH | ユーザーへの通知が必要か判定 |
| メモリ統合 | MEDIUM | メモリの整合性チェック |
| ドリームトリガー | LOW | AutoDream の起動条件を評価 |

### プロセス管理

デーモンは PID ファイルを使って多重起動を防止します。SIGTERM / SIGINT を受信すると、グレースフルシャットダウンを行います。

### プロアクティブアクション

以下の条件で自動的にアクションを起こします:

| 条件 | 閾値 | アクション |
|---|---|---|
| アイドル時間 | 30分 | メモリ統合の提案 |
| メモリ圧迫 | 500行 | AutoCompact の実行 |
| エラー検出 | — | エラーレポートの通知 |

Quiet Hours（静止時間）が設定されている場合、その時間帯の通知は抑制されます。

## AutoDream

AutoDream は、アイドル時にメモリを自動的に統合・最適化する機能です。デーモンから自動起動されるほか、手動実行も可能です。

### 処理フロー

```
1. ログ統合    — 直近のログを読み込み、活動統計とツール使用頻度を集計
2. 矛盾検出    — メモリとトピック間の矛盾をキーワードペアで検出
                 (enabled/disabled, true/false 等)
3. 矛盾解消    — LLM を使って矛盾する記述の解決案を生成
4. インサイト抽出 — パターンやトレンドから洞察を抽出
5. メモリ圧縮   — 統合結果でメモリを更新・圧縮
```

### 設定パラメータ

| パラメータ | デフォルト | 説明 |
|---|---|---|
| `autoTrigger` | true | 自動起動の有効/無効 |
| `idleThresholdMinutes` | 30 | 自動起動までのアイドル時間 |
| `maxDurationMinutes` | 10 | ドリームの最大実行時間 |
| `minSessionsSinceDream` | 3 | 前回ドリームからの最小セッション数 |

## マルチエージェント協調

複数のエージェントが協調してタスクを並行処理する機能です。

### アーキテクチャ

```
Coordinator（調整役）
  ├── Worker A（実行者）
  ├── Worker B（実行者）
  └── Worker C（実行者）
```

**Coordinator** はタスクキューを管理し、アイドル状態の Worker にタスクを割り当てます。

### タスク管理

- **優先度キュー** — HIGH / MEDIUM / LOW の3段階でタスクをソート
- **リトライ制御** — 失敗したタスクは最大3回までリトライ、その後は永続的失敗として処理
- **タイムアウト** — 5分以上応答のないタスクは自動的に再割り当て
- **ハートビート** — Worker は定期的にハートビートを送信。無応答の Worker は「offline」に遷移

## 並列ツール実行

複数のツール呼び出しが互いに依存しない場合、それらを並列に実行できます。

### 依存解決

ツール間の依存関係はトポロジカルソートで解決されます。

```
例: read_file(A) → edit_file(A) は依存あり（直列実行）
    read_file(A) と read_file(B) は依存なし（並列実行）
```

### バッチ実行

依存関係のないツール群はバッチとしてまとめて実行されます。各バッチ内のツールは `Promise.all` で並列実行されます。

## 通知システム

LunaCode は複数のチャンネルで通知を送信できます。

### 対応チャンネル

| チャンネル | 設定方法 |
|---|---|
| コンソール | デフォルトで有効 |
| OS 通知 | macOS: osascript, Linux: notify-send, Windows: PowerShell |
| Pushover | `PUSHOVER_USER_KEY` + `PUSHOVER_APP_TOKEN` 環境変数 |
| Telegram | `TELEGRAM_BOT_TOKEN` + `TELEGRAM_CHAT_ID` 環境変数 |

### 通知の優先度

通知には LOW / MEDIUM / HIGH の優先度が設定されます。設定ファイルで最低優先度を指定することで、重要な通知だけを受け取ることができます。

### Quiet Hours（静止時間）

通知を抑制する時間帯を設定できます。深夜の不要な通知を避けるのに便利です。

```json
{
  "notifications": {
    "quietHours": {
      "start": "23:00",
      "end": "07:00"
    }
  }
}
```

## セキュリティ

### RBAC (ロールベースアクセス制御)

ユーザーにロールを割り当て、各ロールに許可されたツールやアクションを制御します。

### サンドボックス実行

コマンド実行をサンドボックス内で行い、危険な操作を防止します。

デフォルトのルール:
- **拒否**: `rm -rf /`, `mkfs`, フォークボム等
- **許可**: `git`, `npm`, `bun` 等の開発ツール

### 監査ログ

全てのアクション（認証、ツール実行、設定変更等）が監査ログに記録されます。最大10,000件を保持し、CSV / JSON 形式でエクスポートできます。

### Undercover モード

AI エージェントの痕跡を除去する機能です。ログや出力から AI / Claude / OpenAI 等の参照を正規表現で置換し、API キーやメールアドレスを難読化します。

## Buddy モード

17種類のペットキャラクターと対話できるコンパニオン AI 機能です。

### ペットの状態

各ペットは以下の3つの状態値を持ちます:

| 状態 | 範囲 | 時間経過 |
|---|---|---|
| 空腹度 (hunger) | 0 〜 100 | 時間とともに増加 |
| エネルギー (energy) | 0 〜 100 | 時間とともに減少 |
| 幸福度 (happiness) | 0 〜 100 | インタラクションで変動 |

### 感情システム

ペットは状態値に応じて10種類の感情を表現します:

happy, sad, excited, tired, hungry, playful, sleepy, angry, curious, neutral

感情はペットの応答テキスト（日本語）に影響します。

### 応用例

- 開発のモチベーション維持に
- 休憩のリマインダーとして（ペットが空腹になったら休憩）
- チームのマスコットとして
